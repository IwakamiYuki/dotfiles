# コードレビューガイドライン

このドキュメントは `review-diff` スキルのレビュアーエージェントが参照するガイドラインです。

---

## レビューの哲学

### 建設的なフィードバック
- 問題点だけでなく、良い実装も積極的に評価する
- 「なぜ問題か」を具体的に説明する
- 修正案を必ずセットで提示する
- 個人を批判せず、コードを評価する

### 優先度の付け方
- **Critical**: セキュリティ脆弱性、データ損失リスク、重大な性能劣化、アーキテクチャ違反
- **Warning**: 潜在的なバグ、設計の改善余地、テスト不足
- **Info**: スタイル提案、最適化の機会、代替実装の提示

## セキュリティチェックリスト

### OWASP Top 10（2021）
1. **A01: Broken Access Control** — 認可チェックの欠如
2. **A02: Cryptographic Failures** — 弱い暗号化、HTTP（非 HTTPS）
3. **A03: Injection** — SQL/NoSQL/OS インジェクション、XSS
4. **A04: Insecure Design** — セキュアデザインパターンの欠如
5. **A05: Security Misconfiguration** — デフォルト設定、不要な機能
6. **A06: Vulnerable Components** — 既知の脆弱性を持つライブラリ
7. **A07: Auth Failures** — 認証の欠陥、弱いパスワードポリシー
8. **A08: Software Integrity Failures** — 信頼できないソースからのコード
9. **A09: Logging Failures** — 不十分なログ・監視
10. **A10: SSRF** — サーバーサイドリクエストフォージェリ

### 機密情報の扱い
```
# 以下のパターンは即座にフラグ
- API_KEY=... (ハードコーディング)
- password = "..." (ハードコーディング)
- SECRET_TOKEN = "..." (ハードコーディング)
- private_key = "..." (ハードコーディング)
```

## パフォーマンスチェックリスト

### データベース
- N+1 クエリ問題（ループ内でのクエリ実行）
- 適切なインデックスの使用
- 不必要な SELECT * の回避
- トランザクションの適切な使用

### アルゴリズム
- O(n²) 以上の計算量が必要かどうか
- ソートの必要性の確認
- キャッシュ可能な計算の識別

### メモリ
- 大規模データの一括読み込みの回避
- リソースの適切なクローズ（ファイル、DB 接続）
- 無限成長するデータ構造の検出

## コード品質チェックリスト

### 命名規則
- **変数名**: 意図が明確か（`x` より `userCount`）
- **関数名**: 動詞 + 名詞の形（`getUser`, `validateEmail`）
- **定数名**: 大文字スネークケース（`MAX_RETRY_COUNT`）
- **クラス名**: パスカルケース（`UserRepository`）

### 関数設計
- 1 関数 = 1 責任
- 引数は 3 個以下が理想
- 副作用を明確に文書化
- 早期リターンでネストを浅く

### DRY 原則
- 3 回以上の重複は関数化を検討
- コピー&ペーストコードの排除
- 設定値の一元管理

---

## アーキテクチャチェックリスト

### SOLID 原則
- **S**: 単一責任（クラス/関数は 1 つのことをする）
- **O**: 開放/閉鎖（拡張に開き、変更に閉じる）
- **L**: リスコフ置換（サブクラスは親クラスと置換可能）
- **I**: インターフェース分離（不要なインターフェースを強制しない）
- **D**: 依存性逆転（高レベルは低レベルに依存しない）

### 依存関係
- 循環依存の検出
- 外部依存の最小化
- インターフェース経由での依存

### レイヤー分離
- UI 層が Business ロジックを含まない
- Business 層が UI 詳細を知らない
- Data 層が Business ロジックを含まない

---

## テストチェックリスト

### カバレッジ
- 新しいパブリック関数にはテストが必須
- バグ修正には回帰テストが必須
- エッジケース（null, 空配列, 境界値）のカバー

### テスト品質
- テスト名が意図を説明している（`test_should_return_error_when_input_is_null`）
- 1 テスト = 1 アサーション（可能な限り）
- テストは独立して実行可能
- テストデータは最小限で明確

### TDD ルール（プロジェクトで TDD が採用されている場合）
- テストなしの機能追加は Warning
- テストなしのバグ修正は Warning
- 失敗するテストなしのリファクタリングは Info

## プロジェクト固有ルールの読み取り方

レビュー開始時に CLAUDE.md を読み、以下の項目を確認する：

1. **コーディングスタイル** — 言語・フレームワーク固有のルール
2. **コミット規則** — Conventional Commits など
3. **アーキテクチャルール** — プロジェクト固有の設計制約
4. **禁止事項** — "DO NOT MODIFY" などのコメント
5. **テストポリシー** — TDD の採用有無、カバレッジ要件

CLAUDE.md にルールが定義されている場合、それをレビュー基準として優先適用する。

## 差分サイズ別のレビュー戦略

| 差分サイズ | 変更行数 | レビュー深度 |
|-----------|---------|------------|
| 小 | ~50行 | 全項目を詳細にチェック |
| 中 | 50~200行 | 重要項目を重点的にチェック |
| 大 | 200~500行 | Critical/Warning を優先 |
| 超大 | 500行以上 | ファイル単位で分割を推奨 |

差分が 500 行を超える場合は、ユーザーに分割レビューを推奨する。

## 良いコードの評価基準

レビューは問題点の指摘だけでなく、良い実装を評価することも重要：

- **明確な命名**: 意図が一目でわかる変数名・関数名
- **適切なコメント**: 「なぜ」を説明するコメント
- **エッジケース処理**: 境界値・null チェックの適切な実装
- **テストの充実**: 包括的で読みやすいテスト
- **パフォーマンス配慮**: 最適なアルゴリズム選択
